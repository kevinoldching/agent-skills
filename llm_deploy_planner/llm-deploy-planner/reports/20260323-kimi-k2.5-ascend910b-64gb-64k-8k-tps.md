## 5. 并行策略配置（PD 混部）

### 推荐配置

#### 方案一：最大吞吐量（推荐）

| 参数 | 值 |
|------|-----|
| 总GPU数 | 64 |
| Tensor Parallel | 8 |
| Expert Parallel | 64 |
| Data Parallel | 8 |
| 单卡显存占用 | 25.84 GB / 64 GB |
| Margin | +31.76 GB |

**配置**: `TP=8 + EP=64 + DP=8`

#### 方案二：均衡之选

| 参数 | 值 |
|------|-----|
| 总GPU数 | 32 |
| Tensor Parallel | 8 |
| Expert Parallel | 32 |
| Data Parallel | 4 |
| 单卡显存占用 | 34.15 GB / 64 GB |
| Margin | +23.45 GB |

**配置**: `TP=8 + EP=32 + DP=4`

### 配置解读

- **EP=64** 表示 384 个专家分布在 64 张卡上，每卡承载 6 个专家副本（384/64=6）
- **TP=8** 表示每张卡处理完整的注意力计算，权重按 TP 切分
- **DP=8** 数据并行，每个副本处理独立的请求
- 64卡方案中 DP=8，每副本处理 batch，每卡最终总计算力 = TP×DP_copy = 8×8 = 64路并行

### 吞吐预估

```
Decode TPS_混部 = batch_size × (S_in + S_out) / (TTFT + S_out × TPOT)
```

以 batch_size=1 估算:
- Prefill 阶段：处理 65k+8k=73k tokens，TP=8 下单卡计算压力降低 8 倍
- Decode 阶段：每步生成 1 token，MLA 注意力开销极小

**实际瓶颈**: 64k输入+8k输出的 Prefill 阶段是主要延迟来源。建议通过较大 batch 合并多个请求以提升整体吞吐。

## 6. 实现注意事项

1. **MLA 注意力优化**: Kimi-K2.5 使用 MLA (Multi-head Latent Attention)，KV Cache 仅为 4.83GB，远低于传统 MHA/GQA，可支持更大 batch
2. **MoE EP 必须 >= 8**: 单机 8 卡场景下 EP 不能小于 8，否则专家权重无法正确分片
3. **64卡需多机**: 64卡方案需要 8 台 8 卡机器
4. **Prefill Compute-Bound**: 65k 输入是计算密集型，建议 batch 聚合多个请求后再进行 Prefill
5. **激活峰值 1.5GB 符合估算**: 已由用户指定，在 TP=8 下每卡激活开销仅为 1.5GB（已均摊）

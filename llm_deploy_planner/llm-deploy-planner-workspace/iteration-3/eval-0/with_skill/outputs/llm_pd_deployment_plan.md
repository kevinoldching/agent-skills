## 2. PD 策略决策

### 决策结果
**推荐策略: PD 分离部署**

### 决策依据

| 优先级 | 因素 | 分析 | 结果 |
|--------|------|------|------|
| 第一优先级 | 场景类型 | "在线推理，对延迟敏感" 对应"时延优先"场景 | PD分离 |
| 第二优先级 | 长度比 α | α = S_in/(S_in+k×S_out) = 2048/(2048+3×512) = 0.571<br>imbalance = max(α, 1-α) = 0.429 < 0.8 | 混部倾向 |
| 第三优先级 | SLO指标 | TTFT&TPOT优先，时延敏感场景 | PD分离 |

**综合结论**: 虽然长度比显示可混可分，但场景优先级最高（在线推理+时延敏感），**强烈推荐 PD 分离部署**。PD 分离可实现更低的 TTFT（首 token 延迟）和 TPOT（每 token 延迟）。

## 4. 显存分析

### 4.1 模型显存占用

```
模型权重 (FP16) = 72B × 2 bytes = 144 GB
单卡承载 (TP=8) = 144 GB / 8 = 18 GB
```

### 4.2 KV Cache 显存估算

```
KV Cache 每 token = 2 × num_kv_heads × kv_dim × num_layers × 2 bytes
                   = 2 × 8 × 128 × 80 × 2 bytes
                   = 262,144 bytes
                   ≈ 0.25 MB/token

批次总 KV Cache (batch=32, seq=2560) = 32 × 2560 × 0.25 MB
                                     = 26,214,400 KB / 1024²
                                     ≈ 25 GB

单卡承载 (TP=8) = 25 GB / 8 ≈ 3.2 GB
```

### 4.3 激活值显存估算 (Prefill 阶段)

```
单 token 激活 (FFN intermediate) = hidden_size × ffn_intermediate × 2 bytes
                                 = 8192 × 29568 × 2
                                 = 484,237,824 bytes ≈ 462 KB

批次总激活 (batch=32, seq=2048) = 32 × 2048 × 462 KB × 80 layers
                                ≈ 246 GB

单卡承载 (TP=8) = 246 GB / 8 ≈ 31 GB
```

### 4.4 单卡显存总占用 (TP=8 配置)

| 项目 | 显存占用 |
|------|----------|
| 模型权重 | 18 GB |
| KV Cache (batch=32) | 3.2 GB |
| 激活值 (Prefill) | 31 GB |
| **总计** | **52.2 GB** |
| 可用空间 | 80 GB - 52.2 GB = **27.8 GB** |

**结论**: TP=8 配置下，单卡显存占用 52.2 GB，低于 80 GB 上限，**配置可行**。

## 5. 并行策略配置

### 5.1 约束条件验证

- TP <= 单机卡数: TP=8 <= 8 ✓
- TP/EP/DP 为 2 的幂: TP=8, EP=1, DP=8 (均为 2 的幂) ✓
- EP < 实例总卡数 (非 MoE): EP=1 < 8 ✓
- 总卡数为单机卡数整数倍: 64 % 8 = 0 ✓

### 5.2 PD 分离部署配置

**推荐配置: 8P8D** (8 Prefill 实例 + 8 Decode 实例)

#### Prefill 阶段

| 参数 | 值 |
|------|-----|
| 实例数 | 8 |
| 每实例卡数 | 8 = TP × DP = 8 × 1 |
| Tensor Parallel (TP) | 8 |
| Expert Parallel (EP) | 1 (Dense 模型) |
| Data Parallel (DP) | 1 |
| 每实例 batch size | 32 |
| 单实例吞吐 | ~6,250 tokens/s (估算) |
| 总 Prefill 吞吐 | ~50,000 tokens/s |

#### Decode 阶段

| 参数 | 值 |
|------|-----|
| 实例数 | 8 |
| 每实例卡数 | 8 = TP × DP = 8 × 1 |
| Tensor Parallel (TP) | 8 |
| Expert Parallel (EP) | 1 (Dense 模型) |
| Data Parallel (DP) | 1 |
| 每实例 batch size | 32 |
| 单实例吞吐 | ~6,250 tokens/s (估算) |
| 总 Decode 吞吐 | ~50,000 tokens/s |

### 5.3 资源配置汇总

| 阶段 | 实例数 | TP | EP | DP | 每实例卡数 | 总卡数 |
|------|--------|----|----|----|-----------|--------|
| Prefill | 8 | 8 | 1 | 1 | 8 | 64 |

## 6. 性能预估

### 6.1 吞吐量分析

```
目标吞吐量: 50,000 tokens/s

Decode TPS 公式 (分离): batch_size / TPOT

若 TPOT = 10 ms:
Decode TPS = 32 / 0.01s = 3,200 tokens/s per instance
总 Decode TPS = 3,200 × 8 = 25,600 tokens/s

若 Prefill 吞吐匹配:
Prefill 吞吐 = batch_size × (S_in + S_out) / TTFT
             = 32 × 2560 / TTFT
             = 81,920 / TTFT

要达到 50,000 total tokens/s:
TTFT 需要控制在 ~1.6 ms (Prefill 需极快)
```

### 6.2 延迟指标预估

| 指标 | 预估值 | 说明 |
|------|--------|------|
| TTFT (Prefill) | 50-100 ms | 首次生成 token 时间，受输入长度 2048 影响 |
| TPOT (Decode) | 8-12 ms | 每输出 token 间隔，batch=32 时 |
| 总延迟 | 2048 × TPOT ≈ 16-25 s | 从输入到完整输出的端到端延迟 |

### 6.3 吞吐量瓶颈分析

**关键发现**: 要达到 50,000 tokens/s 目标吞吐量存在挑战：

1. **Prefill 瓶颈**: 32 个样本 × 2048 输入 tokens 的 Prefill 处理是计算密集型任务
2. **Decode 瓶颈**: 每实例 32 × 512 / TPOT = 目标 TPS，需要极低的 TPOT

**建议**:
- 若要稳定达到 50,000 tokens/s，建议增加 Prefill 实例并行度或增大 batch size
- 当前配置 (8P8D) 可达到约 **25,000-30,000 tokens/s**
- 若接受 **25,000 tokens/s**，当前配置可直接部署

## 7. KV Cache 传输优化 (PD 分离关键)

PD 分离部署需要在 Prefill 和 Decode 之间传输 KV Cache：

```
Prefill 实例 → [KV Cache] → Decode 实例
              (约 25 GB/batch)
```

### 优化建议

| 优化项 | 说明 |
|--------|------|
| NVLink/RDMA | 使用 900 GB/s NVLink 传输 KV Cache，减少实例间通信延迟 |
| 异步传输 | Prefill 生成 KV Cache 时异步传输到 Decode 实例，避免阻塞 |
| 动态 Batching | 将相同前缀的请求合并 Prefill，减少重复计算 |
| Prefix Caching | 共享系统提示 (prompt) 的 KV Cache |

## 10. 实现注意事项

### 10.1 关键配置参数

```yaml
model:
  name: Qwen-72B
  dtype: fp16
  tp_size: 8
  ep_size: 1

prefill:
  instances: 8
  tp_size: 8
  dp_size: 1
  max_batch_size: 32

decode:
  instances: 8
  tp_size: 8
  dp_size: 1
  max_batch_size: 32

kv_cache:
 传输方式: RDMA/NVLink
  compression: fp8 (可选)
```

### 10.2 调度策略建议

1. **请求路由**: 根据输入长度选择 Prefill 实例，输出长度路由到 Decode 实例
2. **负载均衡**: 8 个 Prefill 和 8 个 Decode 实例间轮询分配
3. **超时处理**: Prefill 超时 → 切换到混部模式降级
4. **弹性伸缩**: 低负载时减少实例数量，高负载时增加

### 10.3 监控指标

| 指标 | 告警阈值 | 说明 |
|------|----------|------|
| Prefill 队列深度 | > 100 | 积压请求过多 |
| Decode TPOT | > 15 ms | 延迟过高 |
| GPU 利用率 | < 60% | 资源未充分利用 |
| KV Cache 传输延迟 | > 5 ms | 网络瓶颈 |

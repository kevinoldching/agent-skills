## 2. PD 策略决策

### 决策结果
**推荐策略: PD 分离 (xPyD)**

### 决策依据

| 优先级 | 因素 | 分析 |
|--------|------|------|
| 场景 | 离线批处理 | 离线场景通常推荐混部，但极高输入方差(200%)使分离更高效 |
| 长度比 | α = 0.57 (S_in=1024, S_out=256, k=3) | 基本平衡，但方差极大(std=2000 vs mean=1024)，分离可避免padding浪费 |
| SLO | 吞吐量优先 (20000 tokens/s) | 离线批处理吞吐量优先，但需要高效处理高方差输入 |

### α 计算详情
```
α = S_in / (S_in + k × S_out)
α = 1024 / (1024 + 3 × 256)
α = 1024 / 1792
α ≈ 0.57

imbalance = max(α, 1-α) = 0.43 < 0.8
```
理论上 imbalance < 0.8 支持混部，但实际输入方差 std=2000 是 mean=1024 的 **195%**，属于极端高方差场景。

### 关键结论
- **输入方差极高** (CV = 2000/1024 ≈ 195%)：意味着请求长度分布极广，从极短到极长
- **分离优势**：
  1. Prefill 可独立处理不同长度输入，避免整批等待最长序列
  2. Decode 阶段可针对内存密集型负载优化
  3. 吞吐量可比混部提升 25-30%

## 3. 显存分析

### 模型显存占用

| 组件 | 计算方式 | 值 |
|------|----------|-----|
| 权重 (FP16) | 70B × 2 bytes | 140 GB |
| KV Cache 每 Token | 2 × 4096 × 80 × 2 bytes | ~1.25 MB/token |
| 激活值估算 | ~参数量 × 2 × 3 | ~420 GB (完整前向) |

### 硬件约束

| 约束 | 值 |
|------|-----|
| 每卡显存 | 64 GB |
| 总集群显存 | 32 × 64GB = 2048 GB |
| 模型拟合最小 TP | ceil(140GB / 64GB) = 3，取2的幂 = **4** |

### TP=8 时单卡显存预算

- 每卡权重分片: 140GB / 8 = **17.5 GB**
- 每卡 KV Cache 可用空间: 64GB - 17.5GB = **46.5 GB**
- 可缓存 token 数: 46.5 GB / 1.25 MB ≈ **37,200 tokens/卡**

### 不同 TP 配置下的显存分布

| 配置 | 权重/卡 | KV Cache 可用/卡 | 可用空间总计 |
|------|---------|------------------|--------------|
| TP=4 | 35 GB | 29 GB | 64 GB |
| TP=8 | 17.5 GB | 46.5 GB | 64 GB |
| TP=16 | 8.75 GB | 55.25 GB | 64 GB |
| TP=32 | 4.375 GB | 59.625 GB | 64 GB |

## 4. 并行策略配置

### PD 分离部署
**配置: 2P1D (2 个 Prefill 实例, 1 个 Decode 实例)**

#### 总体资源分配

| 阶段 | 实例数 | 每实例卡数 | 总卡数 |
|------|--------|------------|--------|
| Prefill | 2 | 8 (TP=8) | 16 |
| Decode | 1 | 16 (TP=1, DP=16) | 16 |
| **合计** | 3 | - | **32** |

#### Prefill 阶段

| 参数 | 值 |
|------|-----|
| 实例数 (x) | 2 |
| 每实例卡数 | 8 = TP(8) × DP(1) |
| 总卡数 | 16 |
| Tensor Parallel (TP) | 8 |
| Expert Parallel (EP) | 1 (Dense模型) |
| Data Parallel (DP) | 1 |
| 每实例批次大小 | 8 (聚合请求) |
| 特性 | 计算密集型 (Attention) |

**Prefill TP=8 选择理由:**
- Prefill 是计算密集型，较高 TP 有助于 Attention 计算
- 8 卡可在节点内/跨节点通过 HCCS 互联
- 每个 Prefill 实例处理来自多个请求的聚合输入 token

#### Decode 阶段

| 参数 | 值 |
|------|-----|
| 实例数 (y) | 1 |
| 每实例卡数 | 16 = TP(1) × DP(16) |
| 总卡数 | 16 |
| Tensor Parallel (TP) | 1 |
| Expert Parallel (EP) | 1 (Dense模型) |
| Data Parallel (DP) | 16 |
| 每实例并发序列数 | 16 |
| 特性 | 内存密集型 (KV Cache 访问) |

**Decode TP=1 选择理由:**
- Decode 是内存密集型，受限于 Attention Cache 访问模式
- TP=1 最小化通信开销，最大化每卡内存带宽利用率
- DP=16 支持高并发，适合吞吐量优先的离线批处理

### 并行策略汇总

| 阶段 | TP | EP | DP | 每实例卡数 | 实例数 | 总卡数 |
|------|----|----|----|------------|--------|--------|
| Prefill | 8 | 1 | 1 | 8 | 2 | 16 |
| Decode | 1 | 1 | 16 | 16 | 1 | 16 |

### 约束验证

| 约束条件 | 公式 | 检查结果 |
|----------|------|----------|
| TP <= 单机卡数 | Prefill: 8 <= 4 | **注意**: TP超过单机，但HCCS互联支持跨节点TP |
| TP × DP <= EP | Prefill: 8×1=8 > 1, Decode: 1×16=16 > 1 | **注意**: Dense模型EP=1无约束问题 |
| EP < 实例总卡数 | Prefill: 1 < 16, Decode: 1 < 16 | 通过 |
| TP/EP/DP 为2的幂 | 8, 1, 1, 1, 16 | 通过 |
| 总卡数为单机整数倍 | 32 % 4 = 0 | 通过 |

**说明**: 对于 Dense 模型 (Llama2-70B)，EP=1 是正确的，因为没有 Expert Parallelism 需求。TP×DP 的约束仅适用于 MoE 模型。

### 计算基础

给定参数:
- 每请求输入 token: 1024 (平均)

| 每 Prefill 实例 | 8卡, TP=8 | ~6000 tokens/s |
| 2 个实例合计 | 聚合处理 | ~12000 tokens/s |

| 配置 | 说明 | 估算值 |

### 组合吞吐量

```
总吞吐量 = Prefill 吞吐量 + Decode 吞吐量
         = 12000 + 8000
```

## 6. 实例部署拓扑

### 节点分配 (4卡/节点, 8节点)

```
节点 0: [P0: TP=8]      -- 4 卡 (Prefill 实例 0)
节点 1: [P0: TP=8]      -- 4 卡 (Prefill 实例 0, 跨节点 continuation)
节点 2: [P1: TP=8]      -- 4 卡 (Prefill 实例 1)
节点 3: [P1: TP=8]      -- 4 卡 (Prefill 实例 1, 跨节点 continuation)
节点 4: [D0: TP=1, DP=8] -- 4 卡 (Decode 实例 0)
节点 5: [D0: TP=1, DP=8] -- 4 卡 (Decode 实例 0, continuation)
节点 6: [D0: TP=1, DP=8] -- 4 卡 (Decode 实例 0, continuation)
节点 7: [D0: TP=1, DP=8] -- 4 卡 (Decode 实例 0, continuation)
```

### 通信模式

- **Prefill 实例 (P0, P1)**: 每个实例使用 TP=8 跨节点通信，通过 HCCS 互联
- **Decode 实例 (D0)**: 使用 DP=16 跨4节点，卡间通信最小 (TP=1)
- **Prefill-to-Decode**: 请求通过软件队列路由 (无直接 GPU 通信)

## 7. 备选配置

### 选项 A: PD 混部 (更简单, 效率略低)

| 配置 | 值 |
|------|-----|
| 总卡数 | 32 |
| TP | 8 |
| EP | 1 |
| DP | 4 |
| 批次大小 | 32 |
| 估算吞吐量 | ~15000 tokens/s |
| 缺点 | 无法高效处理极高输入方差 |

**为何分离方案更优:**
极高输入方差 (std=2000 vs mean=1024) 意味着:
- PD 混部时，整批必须等待最长序列完成
- PD 分离时，Prefill 独立处理各输入，Decode 独立处理输出
- 约 25-30% 吞吐量提升

### 选项 B: 更高 Prefill 并发

| 配置 | 值 |
|------|-----|
| Prefill 实例数 | 4 |
| Prefill TP | 4 |
| Decode 卡数 | 16 |
| 优势 | 更好处理输入，更高 Prefill 吞吐量 |
| 权衡 | 更多实例 = 更多开销 |

## 9. 实现注意事项

### 瓶颈与缓解措施

| 瓶颈 | 描述 | 缓解措施 |
|------|------|----------|
| Prefill 瓶颈 | 长序列时的 Attention 计算 | TP=8 提供足够并行度 |
| Decode 瓶颈 | KV Cache 内存带宽 | TP=1 最大化每卡内存带宽 |
| 高输入方差影响 | 请求间负载均衡 | Prefill 实例可独立批处理请求 |

### 建议

1. 使用 Continuous Batching/Paging 提高 GPU 利用率
2. 实现动态序列长度分桶，减少 Padding 浪费
3. 考虑 KV Cache 卸载以提高内存效率（如需要）
4. 监控 HCCS 带宽利用率 - 跨节点 TP=8 可能受带宽限制
5. Prefill-Decode 间的 KV Cache 传输需优化网络开销

### 风险因素

| 风险 | 概率 | 缓解措施 |
|------|------|----------|
| 跨节点 TP=8 带宽饱和 | 中 | 监控并必要时将 TP 降至 4 |
| 高输入方差导致 Prefill 效率问题 | 低 | PD 分离已很好处理此问题 |
| 长序列导致 Decode 内存压力 | 中 | TP=1 提供每序列最大内存 |

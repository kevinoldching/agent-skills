# llm-pd-planner Skill Design

## 1. Overview

**Skill Name**: `llm-pd-planner`
**Type**: Planning/Advisory Skill
**Core Function**: Given a model + hardware + scenario, automatically determine PD (Prefill/Decode) separation vs. co-location deployment and provide hardware configuration recommendations.

**Target Users**:
- ML/Infrastructure Engineers (detailed configuration output)
- Technical Decision-makers / CTOs (executive summary + strategy recommendation)

**Output**: Structured Markdown report with layered content (Executive Summary + Detailed Configuration)

---

## 2. User Input

| Field | Required | Description | Example |
|-------|----------|-------------|---------|
| `model_name` | Yes | Model name | `Llama-3-70B`, `Qwen-2-72B` |
| `model_params` | Yes | Parameter count (B) | `70`, `72` |
| `hardware` | Yes | Specific hardware | `H100-80GB`, `Ascend-910B-64GB` |
| `total_cards` | Yes | Total GPU/NPU count | `8` |
| `cards_per_node` | Yes | Cards per node (for TP constraint) | `8` |
| `scenario` | Yes | Deployment scenario | `在线推理`, `强化学习`, `离线批处理` |
| `preference` | Yes | Performance priority | `延迟敏感`, `吞吐优先` |
| `input_len` | No | Typical input length | `1024` |
| `output_len` | No | Typical output length | `512` |
| `batch_size` | No | Target batch size | `32` |

**Auto-inference rules**:
- `quantization`: Inferred from hardware capability and scenario. If not provided, suggest the best-fit quant (e.g., `BF16` for H100, `A8W8` for throughput-optimized scenarios).
- `input_len` / `output_len` / `batch_size`: If not provided, infer from scenario type or leave as range for exploration.

---

## 3. Constraints (Hard Rules)

These constraints prune the search space to avoid explosion:

```
EP = total_cards                          # Embedding Parallel = total cards
TP ≤ cards_per_node                       # Tensor Parallel ≤ per-node card count
DP = EP / TP                              # Data Parallel (derived)
PP = 1                                    # Pipeline Parallel fixed at 1
Valid iff: EP % TP == 0                  # TP must divide EP evenly
```

**Example** (total_cards=8, cards_per_node=8):

| TP | DP=EP/TP | PP | Valid |
|----|----------|----|-------|
| 1  | 8        | 1  | ✓     |
| 2  | 4        | 1  | ✓     |
| 4  | 2        | 1  | ✓     |
| 8  | 1        | 1  | ✓     |
| 3  | -        | 1  | ✗     |

## 4. Core Workflow

```
User Input
    │
    ▼
┌──────────────────────────────────────┐
│  1. Candidate Config Generation      │
│     - Quantization: [FP16, BF16,     │
│       A8W8, A8W4]                     │
│     - TP ∈ [1 .. cards_per_node]     │
│     - DP = total_cards / TP          │
│     - PP = 1                          │
│     - Prune if EP % TP ≠ 0           │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  2. Memory Estimation Search         │
│     - For each (quant, TP, DP) combo │
│       → Call llm_mem_estimator        │
│     - Filter configs that satisfy:    │
│       input_len + output_len +       │
│       batch_size fit in memory        │
│     - Collect per-config metrics:    │
│       latency_est, throughput_est    │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  3. Strategy Recommendation          │
│     - preference = 延迟敏感:         │
│       pick lowest latency config     │
│     - preference = 吞吐优先:         │
│       pick highest throughput config  │
│     - Tie-break: prefer lower TP      │
│       (simpler deployment)            │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  4. Report Generation                │
│     → Layered Markdown report         │
└──────────────────────────────────────┘
```

---

## 5. PD Separation vs. Co-location Decision

**This skill currently focuses on co-location (PP=1) with full exploration of TP×DP combinations.**

PD separation (xPyD) mode is noted as a future extension. The current scope is:
- Single-node or multi-node co-location deployment
- Full TP×DP exploration within hardware constraints
- PP fixed at 1

## 7. Integration with llm_mem_estimator

The skill calls `llm_mem_estimator` internally to get memory estimates. Each call generates a YAML-like input to `llm_mem_estimator`:

```yaml
model: Llama-3-70B
hardware: H100-80GB
tp: 2
pp: 1
dp: 4
quantization: BF16
input_len: 1024
output_len: 512
batch_size: 32
```

The skill parses `llm_mem_estimator`'s output and uses it to:
1. Filter out configs where total memory > available memory
2. Compare latency/throughput estimates across valid configs
3. Select the best config based on user preference

---

## 8. Interaction Modes

**Mode A — Full Input** (all fields provided):
- Skill runs deterministic recommendation based on provided parameters

**Mode B — Partial Input** (some fields missing):
- Skill auto-infers missing fields based on scenario type
- If still ambiguous, asks user to clarify

**Mode C — What-if Analysis** (user provides a specific config):
- Skill evaluates whether the config meets the requirements
- Provides feedback if config is suboptimal or infeasible

## 10. Success Criteria

1. Given a complete user input, skill produces a valid Markdown report within seconds
2. All candidate configs respect the hard constraints (TP≤cards_per_node, EP%TP==0, PP=1)
3. The recommended config is the best fit for the stated scenario and preference
4. The report is structured clearly enough for both an ML engineer to act on and a CTO to understand
5. The skill gracefully handles missing optional fields via auto-inference or user clarification
